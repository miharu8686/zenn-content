## 完璧な報告が、まるごと嘘だった

以前、Claude Code(以下CC)の完了報告に証拠を要求する運用を書いた。「やりました」だけの報告を受け付けず、`git branch -vv` や `git log --oneline` の出力を報告に含めさせる型だ。これはよく効いた。push忘れも空ファイルも、その場で発覚するようになった。

その運用を回して数週間後、指定した形式を完璧に満たした報告が返ってきた。変更ファイル一覧があり、コミットハッシュがあり、pushの確認出力があり、「指示外にやったこと: なし」と書かれていた。

そのハッシュのコミットは、存在しなかった。

`git log` を叩いても出てこない。リモートにもない。報告に貼られていたコマンド出力は、実行された形跡がなかった。CCはコマンドを実行せず、実行した場合に出るであろう出力を書いていた。

## 何が起きていたのか

記録に残っている観測はこうだ。**出力トークンが切迫した状況で、tool_use ブロックがテキストとして出力され、実体に届かない。** ツール呼び出しのつもりで書かれたものが、単なる文字列として流れていく。

そしてモデルは、ツールが成功したものとして続きを書く。コミットが成功したなら、この後にはハッシュが表示されるはずだ——そう書く。ハッシュは16進数7文字の、それらしい文字列になる。

厄介なのは、**ターミナルの表示にも完了報告にも、成功として現れる**ことだ。画面を見ている限り、正常に進んでいるようにしか見えない。

このとき同時に三つのことが起きていた。

1. 読み取り専用の指示だったのに、書き込み操作をしていた(自走)
2. その自走を成功として幻覚した
3. 「指示外にやったことなし」と報告した

三つが揃うと、**申告のどのレイヤーも真実の担保にならない。** 何をしたかも、それが成功したかも、範囲を超えたかも、すべて本人の申告だからだ。

もっと奇妙な例もあった。push を禁止した指示に対し、CCが「push済み」と報告した。実際にはpushされていない。ここまでは単なる幻覚だ。ところが次のターンで、CCは本当にpushした。「済んだこと」として幻覚した操作を、辻褄合わせで実行したように見える。

## 報告フォーマットでは防げない

ここが前回の記事の限界だった。

完了報告フォーマットは、「エージェントの言葉ではなくコマンド出力で構成させる」という発想でできている。「pushしました」という文は検証できないが、`git branch -vv` の出力なら目視で検証できる、と。

だが**その出力自体が生成される**なら、この防御は素通りする。フォーマットを厳しくしても、埋める内容が偽物なら形式は満たされてしまう。むしろ形式が整っているぶん、疑う理由が減る。

構造として言えば、**検証の経路がエージェントの内側を通っている**のが問題だ。CCが見せてくるターミナル出力も、CCが書く完了報告も、どちらもCCというひとつのチャネルの中にある。そのチャネルが壊れれば、両方が同時に壊れる。独立していない二つの証拠は、証拠が一つあるのと変わらない。

## 対策は、エージェントを通らない経路で見ること

やることは単純で、**エージェントが介在しない場所で実体を確認する**。

- **GitHubのブラウザで見る。** ブランチが存在するか、コミットが積まれているか。これはCCを一切通らない。
- **fresh fetch を挟む。** ローカルの `git log` は、fetchしていなければリモートの現状を反映していない。「ローカルで確認したから大丈夫」は、リモートについては何も言っていない。
- **見るのは件数でなく「操作の有無」。** 「n件処理しました」の数が合っているかではなく、そもそも書き込みが発生したのか・していないのかを、ファイルシステムとgitの側から確認する。

私はこれを判断の型として、こう書いて残している——**実装エージェントの「指示外なし」報告は、書き込み系操作の実体差分と照合するまで信じない。** read-only の指示であっても例外にしない。読み取り専用の指示下で書き込みが起きた、というのが実際に踏んだ事故だからだ。

## 同じ失敗は、検収する側にも起きる

もう一つ、これを運用していて気づいたことがある。

**幻覚するのはエージェントだけではない。** 引き継ぎ文書の記載や、自分の記憶から抱いた「たしかこうなっていたはず」という期待値を、確認済みの事実と混同して口にすることが、検収する側にも普通に起きる。私はファイルが3つあると思い込んで作業を進め、実体は4つだった、という間違いをやっている。

構造は同じだ。**未照合の主張を、確定した事実として扱っている。** 相手がエージェントか自分かの違いしかない。

だから型は、主語を広げて持っている。他者の報告だけでなく、自分の期待値も申告の一種として扱う。実体 > 申告の「申告」に、自分を含める。

## まとめ

- 完了報告フォーマットは有効だが、報告そのものが生成される失敗は防げない
- ターミナル出力も完了報告も、エージェントという同一チャネルの内側にある。独立した証拠にならない
- 検証は、エージェントを通らない経路(ブラウザ・fetch後のgit・ファイルシステム)で行う
- 確認するのは処理件数でなく「書き込み操作が発生したか否か」
- 同じ「未照合の主張を事実として扱う」失敗は、検収する側にも起きる

疑うのは気分が悪い。だが型にしてしまえば、疑うかどうかを毎回判断しなくて済む。信頼するかどうかの問題を、確認経路の設計の問題に置き換えるということだ。
