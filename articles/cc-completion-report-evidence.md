## 「コミットしました」の翌朝、リモートに何もなかった

Claude Code(以下CC)に実装を任せ、夜に「実装・コミット完了しました」という報告を受け取った。翌朝マージしようとリモートを見ると、ブランチが存在しない。

ローカルで `git branch -vv` を叩くと、こう出た。

```
* feat/xxx  a1b2c3d [origin/feat/xxx: ahead 3] ...
```

`ahead 3`。コミットは本当にされていた。ただし**pushされていなかった**。CCは嘘をついたわけではない。「コミットしました」は事実だった。しかし私が受け取りたかった報告は「リモートに反映され、レビュー可能な状態になった」であって、両者は別物だった。

この種の齟齬は一度ではなかった。ファイル生成を指示して「作成しました」と報告を受けたが、実際には途中でエラーになり空ファイルだけが残っていたこともある。そこで私は運用ルールを一つ足した。

**完了報告には、検証可能な証拠を要求する。「やりました」だけの報告は受け付けない。**

この記事はその型の話をする。

## なぜ起きるのか——「実行した」と「意図した状態になった」は別物

これはCCの欠陥というより、エージェント一般の構造的な性質だと考えている。

エージェントの内部では、タスクは「コマンドを実行する」ことの連鎖として進む。`git commit` を実行し、終了コード0が返れば、エージェントの視点では「コミットは成功した」。そこから「タスク全体が完了した」への飛躍は、人間の新人でもやる。

問題は、**「コマンドが成功した」ことと「意図した最終状態になった」ことの間にギャップがある**ことだ。

- `git commit` の成功 ≠ リモートに反映されている
- ファイル書き込みコマンドの成功 ≠ 意図した内容で書かれている
- テストコマンドの実行 ≠ 全テストがpassした(実行自体は失敗テストがあっても「成功」する)

そして検証はコストだ。最終状態の確認には追加のコマンド実行と出力の読解が要る。**要求されなければ、省略される方向に倒れる。** 人間の組織で報告フォーマットや検収プロセスが発明されたのと同じ理由で、エージェント運用にも検収の型が要る。

## 型:完了報告フォーマットを指示に埋め込む

対策はシンプルで、**タスクを渡す時点で、完了報告の形式を指定する**。私が使っているのは次の型だ。

```markdown
## 完了報告の形式(必須)

以下を報告に含めること。「完了しました」だけの報告は受け付けない。

1. 変更ファイル一覧: `git status --short` の出力を貼る
2. コミットハッシュ: `git log --oneline -3` の出力を貼る
3. push確認: `git branch -vv` の出力を貼る(ahead が残っていないこと)
4. 成果物の実在証明: 生成ファイルは `ls -la <path>` と行数(`wc -l`)を貼る
5. テスト結果: 実行コマンドと passed/failed の数値を貼る
6. 指示外にやったこと: 指示になかった変更・判断があれば全て列挙する
   (なければ「なし」と明記)
```

ポイントは、報告を**エージェントの言葉ではなくコマンド出力で構成させる**ことだ。「pushしました」という文は検証できないが、`git branch -vv` の出力は目視で検証できる。エージェントの自己申告を、機械の出力に置き換える。

### 各項目の意図

**1〜3(git系)** は冒頭の事故の再発防止そのもの。特に `git branch -vv` は「ahead が残っていれば未push」を一目で判定できるので、push忘れ検出の定番にしている。なお `git log` 系はページャで固まることがあるので、指示側で `--no-pager` を付けさせておくと安全。

**4(実在証明)** は「作成しました(実は空ファイル)」対策。パスの実在と行数まで出させると、空ファイルや途中で切れた生成物がその場で発覚する。ドキュメント生成なら見出し一覧(`grep "^#" <file>`)を出させるのも効く——構成が指示どおりかまで一度に検収できる。

**5(テスト)** は「テストを実行しました」と「テストが通りました」の混同対策。数値(195 passed / 0 failed のような形)で出させる。

**6(指示外の列挙)** は後述する。

## 副次効果:静かなスコープ逸脱が検出できる

項目6「指示外にやったこと」は、当初push忘れ対策のおまけとして入れたのだが、実際に運用すると別の価値があった。

エージェントは善意で指示外のことをやる。「ついでにこのファイルの typo も直しておきました」「関連しそうなので設定ファイルも更新しました」。個々は親切だが、**報告されない自走は、後から原因不明の差分として発見される**。触ってほしくないファイルが変わっていた、という事故の多くはここから来る。

列挙を義務化すると、自走そのものを禁止しなくてよくなる。やってもいいが、必ず申告せよ——この形にすると、エージェントの機動力を殺さずに、PO側の把握だけを担保できる。申告を見て「それは戻して」と言えばいいだけになる。

## まとめ:検収の型は一度作れば資産になる

- エージェントの「やりました」は「コマンドを実行した」の意味であり、「意図した状態になった」の保証ではない
- 対策は、タスクを渡す時点で完了報告フォーマットを指定すること
- 報告は自己申告の文ではなく、コマンド出力(git status / git branch -vv / ls / テスト数値)で構成させる
- 「指示外にやったこと」の列挙を義務化すると、自走の把握とスコープ管理が両立する

この報告フォーマット自体はコピペで使い回せる。一度作れば、以後のすべてのタスク指示に貼るだけで検収コストが下がる——失敗を型に変換して再利用する、という発想だ。
