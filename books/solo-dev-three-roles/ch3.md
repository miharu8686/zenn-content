## 破れないルールへの翻訳

第1章で見たとおり、文書に書かれた規律は最も忙しい日に破られます。だからこの章の主題は「よいルールの書き方」ではありません。守るべきルールを破れないルールへ変換する、配線の実物です。

前章の終わりで、こう予告しました。分離を構造に預けるという発想を、もう一段徹底させると、ルールを人が守る対象ではなく、そもそも破れない仕組みに変えるところまで行き着く、と。本章で扱うのは、その一番硬い部品です。設定ファイルに数行の正規表現として書かれ、コマンドが走る前に発火し、当のAI自身にも解除できない、機械的な強制の層。

そしてもうひとつ、本章にはあらかじめ言っておかなければならないことがあります。強制装置は、作って終わりではありません。装置自体を測定し、盲点を記録し、改訂し続ける運用までを含めて、はじめて「強制」と呼べます。フックを書いた日から、フックは信仰の対象になりかねません。「もう禁止したから大丈夫」という一文が、実は何も検査されていない——その落とし穴が、この章の後半の主題になります。

## 規律は文章ではなく exit 2 で語らせる

`~/.claude/settings.json` の PreToolUse フック4本が、この体制の物理法則です。

一本目、`git add -A` / `--all` / `.` の拒否。二本目、heredoc を使ったコミットの拒否。三本目、main への直接 push の拒否。四本目、秘匿ファイル(vault.enc・.env)へのアクセス遮断。いずれもコマンドが実行される前に exit 2 で止まります。事後に気づいて謝る仕組みではなく、事前に不可能にする仕組みです。

なぜこの4本なのか。一本目が禁じる無引数の `git add -A` は、2026年5月、自動pushスクリプトの中で全ファイルを掃き集め、意図しないファイルを巻き込んだコミットを量産しました。「Update: … 他13件」という自動生成のコミットメッセージが、当時の日誌にいくつも並んでいます。しかもそのスクリプトには構造マップ生成というAPI課金を伴う副作用が付いていました。掃除機のスイッチが、請求書と繋がっていたわけです。この事故のあと `add -A` はスクリプトから切り離され(5月29日、`4b9402c`)、やがてフックとして機械強制されるに至ります。二本目が禁じる heredoc も同様で、PowerShell の heredoc を使ったコミット操作は、実際にコミットメッセージを破壊しました。だから今では、コミットメッセージは一時ファイルに書いて `-F <file>` で渡す以外の経路がありません。

この二本は、一般論としての「べき論」ではありません。一度払った授業料の、領収書です。ですが残る二本——main への直接 push の拒否と、秘匿ファイルへのアクセス遮断——は、事故を待たずに置かれた予防的な配線です。そして、事後に謝る仕組みではなく事前に不可能にする、というこの節の主題が最も純粋に表れているのは、領収書のほうではなく、この予防的な二本のほうなのです。

さて、この4本には、注意深く見ないと見落とす二つの設計判断が埋め込まれています。

第一に、拒否メッセージが次の正しい行動を指示していることです。`add -A` を止めるとき、フックは「禁止です」とだけ言うのではなく、「対象ファイルを明示指定してください」と言います。heredoc を止めるときは、「`-F <file>` で渡してください」と言います。止めるだけの壁は迂回を誘発しますが、出口を示す壁は誘導になります。行き止まりに立たされた者は抜け道を探しますが、矢印を示された者はその方向に歩きます。エラーメッセージの一文が、壁を通路に変えているのです。

第二に、ハンドオフの定型(`~/.claude/commands/handoff.md`)に、「フックで機械的に強制。ブロックされたら正常動作なので回避せず従う」と明記されていることです。この一文の意義は、実装役に何を教えるかではなく、何を思わせないかにあります。AIは有能なので、障害物を工夫で乗り越えようとします。コマンドが弾かれれば、弾かれない書き方を探すでしょう。それは、ある意味では優秀さの発露ですらあります。フックが発火したときそれを障害ではなく仕様として受け取らせる一文がなければ、強制装置は知恵比べの相手に格下げされてしまいます。壁を作るだけでは足りず、「この壁は正しい」と伝える必要がある。強制の物理層のすぐ隣に、それを解釈する言語層が要るのです。

## 強制装置そのものが、ground-truth の対象である

フックを書いたら規律が守られる——とは限りません。

このことを実証したのが、`INFRA_INVENTORY_2026-07-05.md` という一本の棚卸し文書です。~/.claude 配下の設定・エージェント定義・フックを全部読み、発火経路を無害なプローブ(echo だけを実行し、状態は一切変えない)で確かめる。それだけの、地味な作業です。ですがそこから出てきた事実は、フックへの素朴な信頼を突き崩すものでした。

第一に、初期の matcher は `Bash` のみを対象にしていました。つまり PowerShell 経由で発行されたコマンドは、4本のフックすべてを素通りしていたのです(旧2-b)。Windows で作業していれば PowerShell は日常の道具ですから、これは「禁止したはずの操作の半分が禁止されていなかった」に等しい状態でした。第二に、`.env` の遮断パターンは、無害な `.env.example` まで誤って止めていました(旧2-d前段)。秘密でも何でもない、リポジトリに commit するのが当たり前のテンプレートファイルが、秘匿ファイル扱いで拒否されていたわけです。そして第三に——ここが重要なのですが——2026年7月6日にこれらを修正したあとでもなお、`git add .\foo.txt` という、PowerShell では完全に慣用的で正当な targeted add が、偽陽性でブロックされることが判明しました(2-h)。パターンの許容文字クラスにバックスラッシュが入っていなかったため、`.` の直後の `\` が「単独のドット + 非許容文字」として拾われてしまうのです。しかも、そのとき出るエラーメッセージは「対象ファイルを明示指定してください」——対象ファイルを明示指定した結果として出ているのですから、これほど紛らわしい案内もありません。

たかだか正規表現1本のフックに、素通り・過剰マッチ・偽陽性という仕様バグの全型が揃って出ました。しかも最後の一つは、初版の棚卸しが「これは該当しないはずだ」と机上で判定していたものを、第2版が echo だけの無害なプローブで実際に叩き、判定の誤りを突き止めたものです。読んだだけでは分からず、走らせて初めて分かった。フックのコードは10行に満たないのに、です。

これが意味するのは、「フックがあるから安全」は「フックの実効範囲を測定したから安全」の略記としてしか成立しない、ということです。フックはコードであり、コードにはバグがあります。テストされていないコードが信用できないのと同じ理由で、プローブされていないフックは信用できません。守らせる仕組みは、守られているかを確かめる仕組みとセットでなければ、安心という名の無防備になります。

第1章で、AIは「現在」を知覚できないと書きました。同じことが、強制装置についても言えます。設定ファイルに書かれた意図と、実際に発火する範囲は、別の存在です。前者を読んで後者を知った気になることが、ここでの推測進行なのです。

## 盲点は消せない。だが記録した盲点は無害化できる

棚卸しは、塞げていない穴も正直に記録しています。

たとえば、main をチェックアウトした状態での裸の `git push`。これはコマンド文字列のどこにも main という語が現れないため、三本目のフックを素通りします(2-g)。フックが見ているのは意図ではなく文字列なのですから、当然のことです。あるいは、Read・Grep ツール経由の秘匿ファイル読み取り。四本目のフックの matcher は Bash と PowerShell だけを見ているので、Read ツールで `.env` を開く行為には何も掛かりません(2-d残)。ここはモデルの自制に依存しています。

重要なのは、これらが「未修正のバグ」としてではなく、「運用規律で塞ぐと決めた残余」として文書化されている点です。裸の push については、「main 上で作業しない」という運用規律が実質の防壁である、と棚卸しは明記しています。技術的に塞ぐ代わりに、人間の運用で塞ぐ。そう決めて、決めたことを書き残しているのです。

なぜ全部を機械で塞がないのか。塞げないからです。機械的強制のカバレッジは原理的に部分でしかありません。文字列を見るフックは意図を見られませんし、ツールを絞る matcher は絞られていないツールを見られません。どこまで正規表現を強くしても、その外側は必ず残ります。だから防衛は階層になります——文書の規律が第一層、フックが最終防衛線、その間を運用習慣が埋める。三層のどれもが単独では不完全で、重ね合わせて初めて厚みになります。

そして、危険なのは盲点の存在ではありません。盲点を知らないまま全面装甲だと信じることのほうです。穴の位置を知っていれば、そこには近づかない、あるいは近づくときは息を止める、という振る舞いができます。ですが自分の装甲に穴がないと信じている者は、穴に向かって全速力で走ります。第1章で見た失敗の多くが、能力ではなく、自分が何を見ていないかを知らないことから生じていたのを思い出してください。

盲点の一覧表は、強制装置の弱さの告白ではありません。強さの根拠です。

## 禁止の反対側に、注入がある

ここまで、フックを「止めるもの」として見てきました。ですが、この体制のフックの半分は、止めるためでなく、配るために使われています。

`settings.json` には、PreToolUse の4本とは別に、SessionStart というイベントに紐づいた一本があります。それが行うのは、拒否ではありません。セッションが開始されるたびに、`git log --oneline -3` と現在ブランチと `git status --short` を実行し、その出力を会話の冒頭へ自動的に注入するのです。実装役が最初の一文字を読むより前に、直近3コミット・今いるブランチ・作業ツリーの汚れ具合が、すでにそこに置かれています。

第1章の洞察1で見たとおり、AIには「現在」を知覚する感覚器がありません。日付も、ブランチも、コミット履歴も、自分から手を伸ばさなければ触れられない。そして最も忙しい日には、その手を伸ばし忘れます。このフックは、その感覚器の義肢です。本人の心がけに期待せず、毎回、機械的に、最初の一口として現在を食べさせる。実際、棚卸しはこう記録しています——セッション開始時の確認について、このフックが実質的にその役割を代替している側面がある、と。

ここで、この章の全体が一つの形に見えてきます。禁止フックと注入フックは、同じ仕組みの表裏です。前者は悪い行動の確率をゼロに寄せ、後者は正しい知覚の確率を1に寄せる。規律を「破れないルール」に変換するのと、まったく同じ手つきで、確認を「忘れられない習慣」に変換しているのです。どちらも、意志の問題を配線の問題に翻訳しています。

そしてこれが、本章を通して繰り返し姿を変えて現れてきたものの、正体です。強制とは罰の体系ではありません。正しい状態が最小エネルギーになるような、地形の設計なのです。

## 結び——地形を測り続ける

第3章で見たのは、規律を配線に変える作業でした。

exit 2 は文章より雄弁で、出口を示す拒否メッセージは壁を通路に変える。ですがその配線自体がコードである以上、素通りも過剰マッチも偽陽性も起こす。だから強制装置は測定の対象になり、測定して見つかった穴は、隠すのではなく一覧表に載せて、運用規律という別の層に引き渡される。そして同じフックの仕組みが、裏返せば、知覚を毎回強制的に配る装置にもなる。

四つの洞察は、結局こう言っています。ルールを構造に預けたなら、次はその構造を疑い、測り、書き残す番だ、と。前章の結び——分離を、意志でなく構造に預ける——に、本章はもう一行を足したことになります。預けた構造もまた、ground-truth の対象である、と。

強制装置を作ることは、規律の終わりではありません。規律の対象が、人間から装置へ移っただけです。地形を設計した者には、その地形を測り続ける義務が残ります。
