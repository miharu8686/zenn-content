# 第4章 STATE運用とセッション引き継ぎ

## 記憶を持たない後任者へ

セッションは必ず終わります。そして次のセッションは、前の記憶を持たない新品として始まります。

これはAI開発の欠陥ではありません。前提条件です。昨日あれだけ議論した設計方針も、三時間かけて潰した仮説も、次に立ち上げたセッションは何ひとつ覚えていません。同じモデルであっても、です。忘れるのではなく、そもそも最初から知らない。毎朝、優秀だが完全な初対面の相手が出社してくる——そういう職場だと思ったほうが正確です。

この前提を受け入れると、STATEファイルの位置づけが変わります。それは作業の「記録」ではありません。記憶を持たない後任者への「引き継ぎ書」です。そして引き継ぎ書の良し悪しは、書いた者の満足ではなく、次に読む者——人でもAIでも——が誤解なく着手できるかで決まります。

前章までで、規律を構造に預け、その構造を測り続ける話をしてきました。本章で扱うのは、その構造を時間を越えて運ぶ器です。どれだけ精巧に地形を設計しても、明日の自分がその地形図を読めなければ、設計はなかったのと同じことになります。

## 良い引き継ぎ書は、自分自身の限界を宣言している

`JURIKA_STATE.md` の冒頭には、こう書いてあります。

> **メモリ・要約は実体に遅れる。** このファイルと git log・実プロセスが食い違ったら、必ず実体を正とする。

文書が、自分より git log を信じろと書いています。これは奇妙な自己言及です。ふつう、文書は自分の正しさを主張するために書かれます。ですが、この一行は自分の敗北条件をあらかじめ宣言している。読者に対して「私は嘘をつくかもしれない、そのときは私を捨てろ」と言っているのです。この自己言及が、STATE運用の核心です。

なぜそこまで自分を疑うのか。STATEは書かれた瞬間から古び始めるからです。書き終えた次のコミットで、もう現実は一歩先に行っています。これを防ぐことはできません。ですが、「古びたときどちらが勝つか」の優先順位——実体 > STATE > 渡された指示——をファイル自身に焼き込むことはできます。同じ順位は `/state-reflect` の定型にも明記されており、そこには「食い違いを見つけたら実体を正とし、食い違いがあった事実もSTATEに記録する」とまで書かれています。負けを認めるだけでなく、負けた記録を残せ、というわけです。

この順位が空文でないことは、実際の事故が証明しています。2026年7月6日、あるブランチは「すでにマージ済み」という前提で扱われていました。ですが `git fetch` して origin の先端を実測すると、未マージだったのです。同じセッションで、実装ポイントの数も、調査時の見積もり「9箇所」から実体「3ファイル5箇所」へ訂正されました。渡された前提もSTATEの記述も、どちらも実体には勝てなかった。だからこそ `/state-reflect` は、「マージ済み」「PR済み」といった状態を書くときは `git fetch` 後の origin 実体で確認してから書け、と命じます。未マージなら「完了・未マージ・PO検収待ち」と、正確な状態で書けと。

さらに JURIKA_STATE には、もう一段の工夫があります。冒頭の使い方の節に、陳腐化した認識の具体的な症状リストが載っているのです。「Claude が古い認識(……)を口にしたら、このファイルで訂正する」——括弧の中には、すでに乗り換えたはずの旧い音声合成エンジンの名前や、とっくに統合された構成要素を別アプリとして語る言い回しなど、実際に観測された誤りが具体的に並んでいます。実際、旧いエンジン名をそのまま出力してしまう誤りは、過去に一度起きています。

これは、風邪の症状表を配るのに似ています。「熱っぽい、喉が痛い——それは風邪だから医者に行け」。同じように、「この単語が出てきたら、それは古い記憶が喋っている」と教えるのです。引き継ぎ書とは情報の缶詰ではありません。鮮度切れの検知手順を同梱した、生鮮品なのです。

## 更新は差分でなく、全文書き直し

state-update スキル(`~/.claude/skills/state-update/SKILL.md`)の手順4は明快です。

> 既存の章立て(プロジェクト定義／確定事項／現在地／宿題トラッキング／次のアクション／決着済み／変更ログ)は維持し、差分ではなく全文を作り直す

一見、無駄に重いやり方です。変わったのは進捗の二行だけかもしれない。それなのに、毎回ファイル全体を書き直せと言う。差分で直せば十秒で済むところを、数分かけて書き下ろせと言うのです。実際、出力は「完成版ファイルの全文。要約や差分では返さない」とまで念を押されています。

ですが、差分更新とは、変更箇所以外の整合性検査を省略する行為です。

ここが肝心なところです。セクション3を直したとき、それと矛盾するセクション7の古い記述は、差分の外側で生き残ります。あなたは3を見ていて、7を見ていない。7が3と食い違っていることに、気づく機会そのものがない。そうして残骸は蓄積し、STATEの中に「新しい真実と古い真実の地層」ができあがります。

そして、それを読まされる次のセッションは、どの層が現在なのかを推測で選ぶことになります。第1章で見た推測進行の入口が、よりによって、推測を防ぐために書かれたはずの引き継ぎ書の中に掘られてしまう。これほど皮肉なことはありません。地層を含んだSTATEは、何も書かれていないSTATEより有害です。空白は疑われますが、断言された古い記述は信用されるからです。

全文書き直しは、この事態への対処です。書き手が一度コストを払うことで、読み手の検証コストをゼロにする——前払いです。全文を書き下ろすとき、書き手は否応なく全セクションを目に通します。7を読めば、3と矛盾していることに気づきます。整合性の検査が、手続きの副産物として強制的に走るわけです。手順が重いのではありません。重さの正体は、省略されがちな検査そのものです。

そして、章立ての固定——プロジェクト定義／確定事項／現在地／宿題トラッキング／次のアクション／決着済み／変更ログ——は、その書き直しを毎回同じ骨格に着地させる型です。白紙から書き直せば、書くたびに構造が揺れます。ですが骨格が決まっていれば、書き直しは創作ではなく、決まった器に現在を流し込む作業になります。毎回作り直すことと、毎回同じ形になること。この二つは矛盾しません。むしろ、後者があるから前者が現実的な手続きになるのです。

## 決定は理由ごと焼き込む。さもないと何度でも蒸し返される

スキルの手順7は、こう指定します。

> 確定事項・決着済みに入れた決定には、理由も一緒に書く(蒸し返し防止)

この一語——蒸し返し——に、AIとの協働の固有問題が凝縮されています。

新しいセッションは、過去の議論を知りません。だから過去に検討して捨てた案を、新鮮な良案として何度でも提案してきます。悪意も怠慢もなく、構造的に、です。しかも、その案はたいてい筋が通って聞こえます。当然です。かつて一度は有力候補になった案なのですから。人間の同僚なら「それは前に試してダメだった」と一言で終わる話に、毎回まっさらな検討が始まってしまう。

実物を見るのがいちばん早いでしょう。JURIKA_STATE の「確定した設計判断(蒸し返さない)」には、こう書かれています。

> WebView2 の透過は不可(DCompアルファ=白箱 / クロマキー=マゼンタ箱 両方失敗)。→ レイヤードウィンドウで実現、で確定。

注目してほしいのは、結論だけでなく、試して失敗した経路が書いてあることです。DComp のアルファ合成を試したら白い箱になった。クロマキーを試したらマゼンタの箱になった。だから透過はウィンドウの外側でやる、と決めた。ここには、実際に費やされた時間の形が残っています。事実、この結論に至るまで透過表現は繰り返し作り替えられ、最終的に打ち切りが確定するまで相応の日数が溶けています。

もし、ここに「WebView2 の透過は使わない」とだけ書いてあったらどうなるか。次のセッションは考えます——なぜだろう、単に試していないだけでは? DComp を使えばいけるのでは? そして同じ白い箱に辿り着きます。理由のない決定は、次のセッションにとって単なる好みに見えるからです。好みは、より良い提案の前で簡単に覆ります。

対照的に、失敗の記録つきの決定は、再実験を不要にします。同じ運用は他の場面にも見られます。一度は本命に据えた表示方式を、前提が変わった翌日に却下したときは、却下理由を三点そろえて記録し「蒸し返さない」と明記されました。透明化のオプションを補助目的で足したらウィンドウ生成そのものが壊れた件は、理由を添えた禁止事項として——負の知識として——書き残されています。

決定の価値は、結論そのものにあるのではありません。そこへ至る探索を、二度と繰り返さなくてよくなることにあるのです。結論は一行ですが、その一行が節約するのは、丸一日かもしれない。理由を書く数分は、その保険料です。

## 書式は美学ではなく、機械との契約である

スキルの手順6は、変更ログの書式を指定します。

> 末尾の変更ログに、行頭が `- YYYY-MM-DD:` で始まる1行を追記する(この書式は開発日誌の自動転記が依存しているため厳守)

書式の指定と、「厳守」という強い言葉。ここだけ読むと、細かい作法にうるさい文書に見えるかもしれません。ですが括弧の中に理由が書いてあります。そして、その理由は本当です。

開発日誌を自動生成する `dev_diary.py` は、STATEの変更ログをこの行頭パターンの正規表現で走査し、その日の記述を拾い上げて日誌へ転記します。ハイフン、半角スペース、四桁の年、ハイフン区切りの日付——この並びに一致しなければ、その行は存在しないものとして扱われます。さらに言えば、同じパターンは「その日に活動があったか」を判定する箇所でも使われています。書式を外した行は、転記されないだけでなく、その日を「何も起きなかった日」に見せることすらある。人間向けの散文の中の、たった1行の書式が、事実上のAPIスキーマとして機能しているのです。

ここで注目すべきは、依存の存在が理由ごとスキルに明記されている点です。

書式規則は、理由を失った瞬間に「うるさい作法」へ堕ちます。なぜその形でなければいけないのか分からないルールは、忙しい日に「たぶん大丈夫だろう」と崩されます。第1章から繰り返し見てきたとおりです。ですが「厳守」の隣に「自動転記が依存している」と書いてあれば、話は変わります。下位モデルも、次のセッションの自分も、これが趣味の問題ではなく互換性の問題だと理解して守ります。守る理由が、権威ではなく因果になるからです。

そしてもう一つ、より大きな含意があります。文書と機械の結合は、避けるべきものではない、ということです。散文に機械が依存しているのは危うい、と感じるかもしれません。ですが危ういのは結合そのものではありません。隠された結合が事故を生むのであって、宣言された結合はむしろ運用を安定させます。この一行は、書式を変えれば何が壊れるかを、変えようとする者の目の前に置いています。壊れうる場所が見えていれば、人はそこを避けて通れる。前章の言葉を借りるなら、これもまた記録された盲点なのです。

## 結び——時間を越えて運ぶ

第4章で見たのは、構造を時間の向こう側へ運ぶ作業でした。

引き継ぎ書は自分の敗北条件を宣言し、実体に道を譲る。更新は差分ではなく全文書き直しで、書き手が検査コストを前払いする。決定には理由を添え、捨てた道を記録して再探索を封じる。そして書式には理由を添え、機械との結合を隠さず宣言する。

四つの洞察は、同じ一つのことを違う角度から言っています。読み手は、記憶を持っていない。だから書き手は、読み手が持っていないものを全部同梱しなければならない——現在との照合手順を、整合性の保証を、捨てた道の地図を、そして書式が誰と契約しているかを。

第1章の結論は、見えないものを補うのは、より賢いモデルではなく、見える場所に置かれた手続きである、というものでした。STATE運用は、その「見える場所」を時間軸へ拡張したものです。今日の自分が見えている場所は、明日の自分には見えません。三時間後にコンテキストを使い果たした自分にも、見えません。

だから引き継ぎ書は、他人のために書くのではありません。明日には他人になっている、自分のために書くのです。
