## 節約ではなく、資源配分

モデルには階層があります。賢く、遅く、高価なもの。そこそこ賢く、速く、安いもの。この使い分けを、多くの人はコストの話として理解します。安いモデルで済むところは安いモデルで、と。

ですが、それは的を外しています。

モデル階層の使い分けは、安いモデルで済ませる節約術ではありません。上位モデルの希少な判断力を「一度しかできない判断」に集中させ、残りを誰がやっても同じ結果になる形まで刻んで下ろす——資源配分の設計です。

節約と資源配分は、似ているようで正反対です。節約は、出費を減らそうとします。資源配分は、限られたものをどこに投じるかを決めます。前者の関心は総額にあり、後者の関心は配置にあります。そして、上位モデルの判断力は総額を気にするような資源ではありません。使うべき場所に使わなければ、余ったところで何の価値も生まないからです。

この章の素材は、実話です。あるアプリケーションの一部機能を退役させ、設定を移行するためのパッチ群を、上位モデルが準備しました。そして、それを小さいコンテキストのモデルへ引き渡した。その引き渡しのために書かれた三つの文書——分割地図、適用手順書、検証生ログ——が、本章の一次資料です。

前章まで、体制の部品を一つずつ組み立ててきました。本章で見るのは、その体制の上を仕事が流れていく様子です。誰が何を判断し、何を下ろし、下ろされた側が何を受け取るのか。

## 上位モデルの成果物はコードではなく、下位モデルの読解コストを圧縮した地図である

引き継ぎ文書の目的宣言が、すべてを言っています。

> 以後のセッション(小さいコンテキストのモデル)が、リポジトリ全体を読まずにこのパッチ群を安全に適用・検証・応用できるようにする

「リポジトリ全体を読まずに」。ここが要です。上位モデルが引き渡そうとしているのは、パッチそのものではありません。パッチを適用するために読まなくてよい範囲の保証です。

実際、文書は4系列のパッチそれぞれについて、「読む必要がある依存ファイルの最小セット」を明記しています。レガシーファイルを削除する系列は、読むべきファイルが最大1個。APIキーを外部化する系列でも、最大2個。しかも、ただ数を書いてあるのではありません。読まなくてよい理由が添えられています。

> 各モジュールの置換は関数内 1-2 行で、**関数名・シグネチャ・空キー時の戻り値(空文字)は不変**のため呼び出し側を読む必要はない。

呼び出し側を読まなくてよい。なぜなら、呼び出し側から見た振る舞いが何ひとつ変わらないから。この一文が、下位モデルの読解範囲から、呼び出し元の全ファイルを消し去っています。

さらに文書は、参考ファイルにわざわざ「(参考・読まなくてよい)」という札を貼っています。関連はしている。見れば理解は深まる。だが読まなくてよい。上位モデルは、関連するものを列挙したのではなく、関連するもののうち何が不要かを判定したのです。

巨大なコードベースを持つリポジトリで、下位モデルが読むべき範囲を1〜3ファイルまで絞り込む——この圧縮こそが、上位モデルにしかできない仕事です。

パッチを書くだけなら、中位モデルでもできます。指示された変更を、指示された場所に入れる。それは難しい仕事ではありません。ですが、全体を読み、依存を確定し、「ここ以外は見なくてよい」と保証する判断は、全体が頭に入る者にしかできません。保証には責任が伴います。「読まなくてよい」と言い切るには、読んだ上で不要だと確認していなければならない。

知性の階層利用とは、上位が下位の代わりに考えることではありません。下位に考える必要がなくなるまで、問題を刻んでやることです。

## 引き継ぎの通貨は「やった」という語りではなく、再検証可能な証拠である

三つの文書のうち、検証生ログには、主張がほとんどありません。

あるのは、pytest の生出力です。

```
=== branch: work / HEAD: 1ca4655 ===
210 passed, 2 warnings in 3.09s
```

あるのは、import 元がゼロであることを証明する `git grep` の実行結果と、その exit code です。

```
$ git grep -nE "import\s+..." -- "*.py" "*.pyw" "*.ps1" ...
(出力なし)
exit=1   ← ヒット 0 件
```

ヒットゼロのとき `git grep` は exit code 1 を返す。だから「出力がなかった」だけでなく「exit=1 だった」と書く。検索が実行されたことと、その結果が空だったことは、別の事実だからです。

あるのは、push をしていないことの証明です。`git log --branches --not --remotes` を打つと 16 件が返る。ローカルにあってリモートに存在しないコミットが16件——つまり、このスパイクで作られた全コミット(パッチ15個とマージ1個)が、一つ残らずリモートに届いていない。「push していません」という宣言ではなく、push されたものがゼロであることを、コマンドの出力が示しています。

あるのは、スパイク外のリポジトリを汚していないことの証明です。隣接する作業ツリーそれぞれで `git status --porcelain` を採取し、出力なし——clean であることを記録する。

そして、ここが白眉なのですが、隣接する作業ツリーの一つには、無関係な未コミット変更が一件見つかりました。ふつうなら、黙って通り過ぎたくなる場面です。ですが検証ログはこう書いています——この変更は本セッション由来ではない。ファイルの mtime が2026年5月30日、本作業の1ヶ月以上前である。差分の内容も、過去セッションの許可リスト追記3行のみ。

自分の無実を、タイムスタンプで証明している。

なぜ、ここまでやるのか。語りは引き継げないからです。

「テストは通りました」という文は、次のセッションにとって検証不能な伝聞です。通ったのかもしれないし、通ったと思い込んだのかもしれない。何件通ったのか、どのブランチで、いつ。その文からは何も分かりません。ですが、実行コマンドと生出力のペアは違います。ブランチ名と HEAD のハッシュが添えられた `210 passed, 2 warnings in 3.09s` は、いつでも再実行して突き合わせられます。疑わしければ、自分で打てばいい。

第1章で見たとおり、文章の流暢さと情報の確度を、モデルは区別できません。よく書けた報告と、正しい報告の見分けがつかない。だから上位から下位への引き継ぎは、流暢さを削って証拠の密度を上げる方向に書かれなければなりません。

信頼は語りではなく、再現手順の形で相続されるのです。

## 成功手順だけの引き継ぎは、引き継ぎではない

引き継ぎ文書の各系列には、「失敗時の切り戻し」の欄があります。そこには、抽象的な心構えではなく、revert すべきコミットのハッシュが、順番どおりに並んでいます。

適用手順書のほうには、さらに踏み込んだ記述があります。cherry-pick がコンフリクトした場合、どうするか。

> 解消せず `git cherry-pick --abort` して、dev がスパイク時点から進んだ差分を確認する。

解消するな、と書いてあります。中断して、前提が変わっていないかを確認せよ、と。コンフリクトは解くべきパズルではなく、前提のズレを知らせる信号だからです。下位モデルにその場でコンフリクトを解かせれば、地図の外側で判断させることになります。地図の外に出そうになったら、止まって戻れ。

そして、もう一つ。文書には【PO検収時の明示確認事項】という札の付いた項目があります。

APIキーの取得優先順が、設定ファイル優先から環境変数優先へ反転する。コードとしては意図どおりで、テストも全緑です。ですが、もし環境に古い、あるいは別用途のキーが環境変数として設定されていれば、従来は設定ファイル側が勝っていたところで、これからは環境変数側が使われます。技術的には正しい。にもかかわらず、人間の明示的な了解が要る。だから、機械的な検証項目とは区別して、印が付けられています。

これは悲観ではありません。権限設計です。

下位モデルが安全に自走できる範囲は、失敗しても切り戻せる範囲と正確に一致します。切り戻せる場所では、思い切ってやらせてよい。切り戻せない場所では、やらせてはならない。境界はそこに引かれます。切り戻し手順のないタスクを下ろすことは、権限だけ渡して保険を渡さないことに等しいのです。

そして、「技術的には正しいが人間の判断が要る変更」を、機械的な検証項目と区別して明示すること。これは第2章で見た、PO専管事項の境界を、パッチという最下流の実装物にまで貫通させる行為です。事業の判断、設計の判断、実装の判断——そのどれもが、同じ形で「ここから先は人間」と書かれている。

判断の階層は、文書の階層のすべてで同じ形をしていなければなりません。

## 仕事を刻む単位は、モデルの生成能力ではなく人間の検収帯域で決める

適用手順書の指示は、頑なです。

> 各系列で共通の流れ:「ブランチ作成 → cherry-pick → pytest → push → PR → 人間がマージ → worktree を dev 最新に進めて次系列」。**系列をまとめて1PRにしない**(切り戻し単位を保つ)。

1系列 = 1ブランチ = 1PR。この単位で人間が検収する。

奇妙に思えるかもしれません。技術的には、4系列に依存はないのです。任意の順で適用できます。まとめて一度に入れても壊れません。それは推測ではなく、実証済みです——4系列を統合したブランチを一時的に作り、順に `--no-ff` でマージしたところ、全マージがコンフリクトなしで自動解決し、pytest は 236 件全緑でした。期待値(210 + 3 + 9 + 14 = 236)と実測が一致しています。

つまり、まとめて1PRにできることは分かっている。それでも、あえて4回に分けて、4回とも人間の検収を通す。

推奨される適用順すら、技術的な理由では決まっていません。「1 → 2 → 3 → 4(リスクの小さい順)」。依存関係の順ではなく、人間が安心して飲み込める順です。最初に、どこからも参照されていないファイルの削除。最後に、キーの取り扱いという神経を使う変更。技術ではなく、人間側の都合が順序を決めています。

ここに、階層運用の律速資源が露出しています。

AIの生成能力は、事実上無限になりました。パッチは一晩で15個できます。テストも一緒に書けます。ですが、人間が差分を理解し、責任を引き受けられる速度は、変わっていません。四つの変更をまとめて見せられた人間は、そのうち一つしか本気では読まないでしょう。そして残り三つに、責任だけを負うことになります。

一人開発のスループットを決めるのは、今やモデルの速さではありません。検収の帯域です。

だから仕事は、「AIが一度に作れるサイズ」ではなく「人間が安全に飲み込めるサイズ」で刻みます。第1章の洞察3——同時変更の組合せ爆発——と同じ非対称が、ここではプロセス設計の原理として現れています。あのときは、絡み合う変更を同時に入れたせいで原因が特定できなくなりました。ここでは、絡み合わない変更ですら、あえて分けています。理由は同じです。切り分けられる状態を、事前に作っておくためです。

生成が速くなるほど、刻む規律の価値は上がるのです。

## 結び——刻むことが、上位の仕事である

第6章で見たのは、仕事を階層に流す設計でした。

上位モデルが残したのはコードではなく、読まなくてよい範囲を保証した地図でした。引き継ぎの通貨は語りではなく、再実行できるコマンドと生出力でした。下ろされた仕事には切り戻し手順が添えられ、人間の了解が要る変更には札が貼られていました。そして刻む単位は、生成の速さではなく、検収の帯域で決められていました。

四つの洞察は、同じ一つの姿勢を指しています。上位の仕事とは、答えを出すことではなく、答えを出す必要がない状態を作ることだ、と。

考えてみれば、この本のここまでの章も、すべて同じことを言ってきました。第3章のフックは、正しい判断を毎回下さなくて済むように、間違いを不可能にしました。第4章のSTATEは、記憶に頼らなくて済むように、現在を器に流し込みました。第5章の出力形式は、確認の質を意志に委ねなくて済むように、書式で縛りました。

そして第6章の分割地図は、下位モデルが全体を理解しなくて済むように、範囲を保証しました。

体制とは、賢さを増やす装置ではありません。賢さを必要とする場所を減らす装置です。そして、どこで賢さを節約できるかを見抜くこと——それだけが、節約できない仕事なのです。
