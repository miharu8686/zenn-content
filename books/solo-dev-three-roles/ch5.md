# 第5章 ground-truth検証

## 確認を、測定に置き換える

第1章の洞察1で、AIには「現在」を知覚する感覚器がない、と述べました。この章は、その処方箋の実装です。

核心は一つです。「確認する」という曖昧な行為を、出力形式が固定された「測定」に置き換えること。

この二つの言葉の距離を、まず感じてほしいと思います。確認は、心の中で起きます。誰にも見えず、どこにも残らず、やったかどうかは本人の申告に頼るしかありません。しかも本人ですら、どこまで見たかを正確には覚えていない。一方、測定は形式を持ちます。数値なり、ハッシュなり、埋めるべき欄がある。埋まっているか空欄かは、外から見て分かります。

確認は気分で薄まりますが、測定は形式が守るのです。

前章まで、規律を配線に変え、それを時間の向こう側へ運ぶ話をしてきました。ですが運ばれてきた引き継ぎ書が、着地した瞬間に現実とズレていたら、どうなるか。第4章のSTATEは「自分より実体を信じろ」と書いていました。本章で扱うのは、その「実体を信じる」を、気合ではなく手続きとして実行する装置です。

## 出力を2形式に固定すると、「なんとなく確認した」が存在できなくなる

`/ground-truth` コマンド(`~/.claude/commands/ground-truth.md`)の出力仕様は、2つしかありません。

> 出力は2形式のみ:
> - 「実体確認OK (HEAD `<hash>` / branch `<name>` / status要点1行 / worktree・プロセスの要点)」
> - 「食い違いあり: `<前提>` ↔ `<実体>`」の列挙

第三の形式はありません。「たぶん大丈夫です」も、「特に問題はなさそうでした」も、この定型のどこにも置き場所がないのです。

この設計の狙いは、確認という行為の最大の敵——曖昧な完了——の排除にあります。

「見ました、大丈夫そうです」は、確認のふりをした無為です。恐ろしいのは、それが嘘ではないことです。本人は本当に見たのかもしれない。ただ、何をどこまで見たのかが本人にも分かっていない。そして受け取る側は、その一文から何も検証できません。確認したという事実の代わりに、確認したという気分だけが受け渡されます。

ところが、出力形式を固定すると、この逃げ道が塞がります。OK と答えるには HEAD のハッシュを埋めねばなりません。ハッシュを埋めるには、`git log` を実際に叩くしかない。ブランチ名の欄を埋めるには、ブランチを見るしかない。つまり、出力形式が作業の実施を強制するのです。書式が、行為を裏側から規定しています。

さらに、この定型は埋め草の材料まで指定しています。

> 根拠は必ず実体(ハッシュ・ブランチ名・ファイル名・PID)で示し、推測で埋めない。

ハッシュ、ブランチ名、ファイル名、PID。いずれも、実体に触れなければ手に入らない値ばかりです。想像で書けば、次の一手で必ず矛盾します。存在しないハッシュは `git show` で消えますし、でっちあげた PID はプロセス一覧に現れません。形式が、嘘に対して自動的に脆いように作られている。

測定とは、サボり方が構造的に存在しない確認のことなのです。

## 測るのは世界そのものではなく、認識と世界のズレである

ここで、この定型の重心がどこにあるかを見誤らないようにしたいと思います。

`/ground-truth` の本体は、`git log` を打つことではありません。手順の末尾にある、この一行です。

> 前提との突き合わせ: $ARGUMENTS
> (引数・STATEファイル・作業指示で HEAD・ブランチ・working tree の状態などの前提が渡されている場合、実体と突き合わせる)

渡された指示やSTATEが主張する前提——HEADはこれ、ブランチはこれ、進捗はここまで——と、実体を突き合わせ、ズレを検出する。それがこのコマンドの仕事です。

なぜ、突き合わせが本体なのか。`git log` 単体は、事実の羅列にすぎないからです。ハッシュが並んでいるのを眺めても、そこには何の危険もありません。事故を起こすのは事実ではなく、事実と認識の乖離のほうです。「マージ済みのはずだ」という認識と、未マージという事実。その二つが同じ部屋にいることに気づかないまま作業を始めると、下流のすべてが崩れます。実際、第4章で見たとおり、「マージ済み」という前提は `git fetch` の一発で覆されました。前提そのものが間違っていたのではなく、前提が更新されていなかったのです。

だから測るのは、世界そのものではありません。認識と世界のズレです。物差しを当てる先は git ではなく、git と自分の頭の間の隙間なのです。

確認対象の選定にも、同じ思想が通っています。

このコマンドが見にいくのは、`git log` と `git status` だけではありません。`git branch -vv` で追跡ブランチと ahead/behind を見て、`git worktree list` で「どの worktree がどのブランチをロックしているか」を確認し、そして関連プロセスの実在を確認します。プロセス確認の項には、こう添えられています——「停止はしない。稼働中プロセスは旧コードを保持している点に留意」。

この一文の重さが分かるでしょうか。コードを直した直後の開発者の認識は「直った」です。ですが、再起動していないプロセスの中では、旧コードが今も動いています。ファイルの上では修正済み、メモリの上では未修正。同じソフトウェアが、二つの異なる状態で同時に存在している。ここは認識と実体が乖離する典型点です。だからチェックリストに載っています。

worktree の項も同じです。複数の worktree を持つリポジトリでは、編集したつもりの場所と、実際に編集が落ちた場所が食い違いえます。これは架空の心配ではなく、実際に起きています——編集が意図したブランチの作業ツリーではなく、別の worktree に着地し、あとから正しい側へ再適用する羽目になったことがありました。以来、どの worktree がどのブランチをロックしているかは、着手前に確認すべき項目として明文化されています。

つまり、ground-truth の項目とは、過去に乖離が実際に起きた場所の地図なのです。抽象的な網羅性から導かれたチェックリストではありません。誰かが一度そこで転んだから、そこに杭が打たれている。チェックリストの各項目は、認識が実体からずれる典型点を、一つずつ指しているのです。

## 検証者から修正権限を奪うことが、検証の純度を守る

第2章で、サブエージェントは全員が読み取り専用の専門職だと書きました。本章では、その設計が検証という行為にとって何を意味するのかを見ます。

検証系のサブエージェント——ground-truth-check、explorer、contrast-debugger、test-runner、code-reviewer、pre-merge-check——は、6体全員が読み取り専用です。定義ファイルには、いずれも変更系コマンドの禁止が明記されています。explorer は「ファイルの作成・編集は一切しない」。test-runner は「修正提案は1行まで。修正自体は絶対にしない」。code-reviewer は「ファイルの書き換え(提案のみ)／git変更系コマンド」を禁じられ、pre-merge-check は報告するだけです。

なかでも、この思想が最も先鋭に現れているのが contrast-debugger です。定義の冒頭は、こう始まります。

> あなたは対照実験専門の調査エージェントです。目的は「直すこと」ではなく「原因の層を1回の調査で確定させること」です。

そして禁止事項には、こうあります。

> 修正の適用(提案は1行まで可)／対照実験なしで仮説修正案を並べること／ファイル変更・git変更系コマンド。

「対照実験なしで仮説修正案を並べること」までが禁止されている点に注目してください。手を出すことだけでなく、手を出したくなる形で考えることまでが封じられています。

なぜ、直せる者に調べさせないのか。

修正権限を持つ調査者は、直せそうな仮説を無意識に優遇するからです。第1章の洞察4で見たとおり、もっともらしい仮説は無料で無限に湧きます。そこに「思いついた修正をすぐ試せる」権限が加わると、調査は仮説の検証ではなく、仮説の消化試合になります。一本潰しては次を思いつき、また潰す。手は動き続け、時間だけが溶けていく。

これは比喩ではありません。ある描画不具合の調査では、レンダラ側の仮説が6本立てられ、順に潰され、6本とも外れました。丸1日が消えました。最後に、正常に表示される公式のサンプルとの対照実験を一度おこなったところ、原因はレンダラ側ではなくデータ側にあると、一発で確定しました。最初に対照していれば、丸1日は数十分だったのです。仮説が足りなかったのではありません。仮説が多すぎたのでもありません。対照という手続きを踏む前に、修正できる場所を探し始めたことが、原因でした。

読み取り専用という制約は、能力の制限ではありません。調査結果から利害を取り除くための、絶縁体です。原因を探す手と原因を直す手が同じであるかぎり、人は自分が直せる場所に原因を見つけたがります。ならば、手を分けるしかない。

検証と変更を、同じ手に持たせない。これは、会計監査が経理を兼務しないのと同じ理屈です。

## 規律の運用実態そのものも、ground-truth の対象である

最後に、この章で組み立ててきた道具を、体制それ自身に向けます。

第3章で参照した棚卸し文書、`INFRA_INVENTORY_2026-07-05.md` の発見1-b は、体制が自分自身を測った記録として読めます。

規定はこうです。共通規律(`~/.claude/CLAUDE.md`)のサブエージェント標準フローは、「セッション開始/STATE受領 → ground-truth-check」と定めています。ではその実運用はどうか。棚卸しが記録した実体は、こうでした——メインセッションが inline で確認しており、当該エージェントの呼び出し実績はゼロ。

規定された手順が、一度も使われていなかった。

ただし、話はここで終わりません。棚卸しは同時に、こうも記録しています。チェック内容自体は毎回行われており、欠落はない。そして、第3章で見た SessionStart フックの自動注入が、その役割を代替している側面がある、と。つまり、規定は守られていないが、規定の目的は達成されている。奇妙な、しかしよくある状態です。

ここで重要なのは、結論ではありません。手つきです。

この規定と実運用のズレを、棚卸しは隠しませんでした。正当化もしませんでした。「フックがあるから実質同じです」と丸めることもしませんでした。代わりに、こう書いています——(a) 規定を実態に合わせて緩める、(b) 規定どおり厳守する。この二択が未決である、と。

未決を、未決のまま文書に固定する。これは、決められなかったことの記録ではありません。決めるべき問いが存在することの記録です。

規律は、書いた瞬間から実態と乖離し始めます。第4章のSTATEが実体から乖離するのと、まったく同じ力学で。文書は静止し、運用は動き続けるからです。だから ground-truth は、git の状態だけに向けられるものではありません。規律自身の運用状態にも、定期的に向けられます。

自分を測れない体制は、形骸化を検知できないのです。

## 結び——測定を、体制の内側にも向ける

第5章で見たのは、確認を測定に変える作業でした。

出力を2形式に固定すれば、曖昧な完了は座る椅子を失う。測定の照準は世界ではなく、世界と認識のズレに合わせられる。検証者からは修正権限を取り上げ、調査から利害を絶縁する。そして最後に、その測定の照準は、体制自身の運用実態へも向けられる。

四つの洞察を貫いているのは、確認は意志では守れない、という一点です。第1章から繰り返してきたことが、ここでもう一度、別の顔で現れました。規律は最も忙しい日に破られる。ならば確認もまた、最も忙しい日に薄まります。「見ました、大丈夫そうです」は、疲れた夜にこそ書かれるのです。

だから確認を、書式で縛る。ハッシュの欄を作り、埋めなければ完了と言えないようにする。第3章のフックが行動を配線に変えたように、第5章の出力形式は、確認を測定に変えています。どちらも、意志の問題を形式の問題へ翻訳する、同じ手つきです。

そして体制は、その物差しを最後に自分自身へ向けました。規定と実運用のズレを測り、未決を未決として書き残した。ここに、この本が扱ってきた体制の性格が、最もよく表れていると思います。

強くなるとは、間違えなくなることではありません。間違えていることに気づける状態を、保ち続けることです。
